Bloo Crucible

v1.0 GA 22 June 2026

Authentication & Session Management

Crucible's authentication layer ensures every investigation, triage action, and escalation is identity-bound and tenant-scoped. All operations require an authenticated session; unauthenticated requests are rejected at the gateway.

Tenant-Scoped SessionsV1.0

Sessions are scoped per tenant and user, with short-lived tokens refreshed transparently. Every triage action, playbook trigger, and escalation is attributed to a specific authenticated identity in the audit record.

Integration Credential StorageV1.0

All integration credentials (ticketing, notification, LLM keys) are sealed in HashiCorp Vault at {tenant_id}/{scope_id}/integrations/{name} the moment they are entered. They are never stored in the application database and never returned by any read API.

Signal Source Configuration

Crucible monitors Datafabric streams as signal sources, continuously evaluating new events against active playbooks. Each scope can have multiple signal sources with independent monitoring cadences.

Datafabric Stream MonitoringV1.0

Administrators configure signal sources by pointing Crucible at one or more Datafabric stream identifiers and setting a polling interval. Crucible queries the stream for new events on each tick and routes matching events to the appropriate playbook pipeline.

Configurable Monitoring IntervalsV1.0

Each signal source has an independently configurable polling interval — high-fidelity sources like authentication logs can be polled frequently while lower-volume sources run on longer cycles to manage cost and load.

Integration Connections

Crucible ships with native integrations for ticketing, notification, and communication platforms. Playbook actions reference a named integration connection — swapping the underlying platform requires no playbook edits.

JiraV1.0

Bidirectional Jira integration for ticket creation, status updates, and comment append. Playbook escalation actions create fully-attributed Jira issues with the investigation summary, evidence, severity, and recommended actions pre-populated.

ServiceNowV1.0

ServiceNow incident and change management integration. Escalated Crucible investigations create ServiceNow incidents with standardised field mapping — priority, category, assignment group, and work notes sourced from the playbook configuration and the AI triage output.

Slack & Microsoft TeamsV1.0

Configurable webhook-based notification to Slack channels or Microsoft Teams channels. Escalation notifications include severity, investigation title, key evidence summary, and a direct link to the Crucible investigation view.

PagerDutyV1.0

PagerDuty event routing for on-call escalation. Critical and high severity investigations trigger PagerDuty incidents routed to the configured service and escalation policy, with deduplication keys tied to the Crucible investigation ID.

Playbook Engine

Playbooks define the conditions under which Crucible triggers an automated investigation. A playbook binds a signal source, a set of detection conditions, a Datafabric query template, and a response configuration into a single reusable unit.

Condition-Based TriggeringV1.0

Playbook conditions evaluate each incoming signal event against field-level filters (field equals / contains / matches regex / threshold exceeded). Events that satisfy all configured conditions trigger the playbook's investigation pipeline.

Reusable Query TemplatesV1.0

Each playbook carries a parameterised Datafabric query template. When the playbook fires, field values from the triggering event are injected into the template, producing a targeted query scoped to the exact entity or time window of interest.

Cooldown & DeduplicationV1.0

A per-playbook cooldown window prevents the same signal from spawning duplicate investigations. Deduplication keys are derived from configurable event fields, suppressing noise while ensuring distinct events are always investigated.

Multi-LLM Provider Support

Crucible's AI pipeline supports four LLM provider families. Each scope can independently select its LLM provider and model, and switching providers requires no pipeline code changes.

AnthropicV1.0

Long-context reasoning for complex multi-stream investigations. Recommended for deep triage tasks that require synthesising evidence across many Datafabric streams.

OpenAI & AWS BedrockV1.0

OpenAI for broad enterprise familiarity and flexible model selection; AWS Bedrock for AWS-resident deployments requiring data residency commitments.

Ollama (Self-Hosted)V1.0

Fully self-hosted, air-gapped environments. No external AI API calls — LLM execution stays entirely within the customer environment. Required for compliance regimes that prohibit outbound data transmission.

AI Triage Agent

When a playbook fires, the Triage Agent autonomously queries Datafabric, evaluates the results, and produces a fully-structured investigation record with severity rating, evidence cards, recommended actions, and complete chain-of-thought reasoning — all without analyst input.

Autonomous Investigation PipelineV1.0

The Triage Agent runs a multi-step pipeline: data collection, result summarisation, severity assessment (critical / high / medium / low / info), evidence structuring, recommended action generation, and chain-of-thought capture. Each step is logged to the audit trail.

Evidence CardsV1.0

Every investigation includes source-attributed evidence cards — each card cites the Datafabric stream, the exact query executed, the result rows, and the AI's interpretation. Analysts can verify every inference against raw data without leaving the investigation view.

Auditable Chain of ThoughtV1.0

The complete chain-of-thought reasoning for every triage step is stored in the investigation record and surfaced in the UI. Analysts and compliance teams can inspect exactly why the AI reached a given severity rating or recommendation.

Escalation & Ticket Creation

Investigations that exceed a configured severity threshold, or that analysts manually escalate, are automatically routed to connected ticketing and notification platforms. The complete investigation record travels with the ticket.

Automatic Severity-Based EscalationV1.0

Playbooks define an escalation severity threshold. When the Triage Agent assigns a severity at or above that threshold, the investigation is automatically escalated — creating a ticket and sending notifications without analyst intervention.

Manual EscalationV1.0

Analysts can manually escalate any investigation regardless of severity, selecting the target integration and overriding the priority if required. Manual escalations are attributed in the audit trail with the analyst identity and timestamp.

Rich Ticket ContextV1.0

Tickets include: investigation title and severity, AI-generated summary and recommended actions, all evidence card content, the chain-of-thought reasoning, and a deep link back to the Crucible investigation view — eliminating context-switching for the responding analyst.

Deployment & Security

Enterprise-grade security,
on your infrastructure.

Both Bloo products are self-hosted on Kubernetes on AWS or Azure. Customer data — including investigation records and AI call content — never leaves your environment.

Self-Hosted KubernetesV1.0

Both products deploy on Kubernetes on AWS or Azure. Customer data — including investigation records, query payloads, and AI call content — does not leave the customer's environment.

Air-Gapped DeploymentsV1.0

Air-gapped deployments are supported via Ollama for LLM execution. No external AI API calls required — the entire triage pipeline runs within the customer perimeter.

Secrets & AuditV1.0

All integration and LLM credentials are sealed in enterprise-grade secrets vaults. Session tokens use short TTLs with silent refresh. Every action and AI reasoning step is written to an append-only audit trail that cannot be modified or deleted.
