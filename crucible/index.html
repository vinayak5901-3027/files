Triage Agent

The Triage Agent is the single owner of the full post-playbook investigation pipeline. It orchestrates all specialist Investigation Agents, aggregates their findings into a confidence-weighted threat score, and produces the final verdict and routing action.

Triage Verdict — Active, Benign & FalseV1.0
  • Active — A genuine, live threat requiring investigation or response. High confidence → escalate immediately.
  • Benign — Real activity but not a threat. High confidence → close automatically.
  • False — Detection misfired; no corresponding real-world event. Low confidence → route to human review.
  • Structured Output — Every run produces: verdict (Active/Benign/False), confidence (0–100%), action (respond/dismiss/escalate), reasoning (human-readable explanation).

Signal Source

Single source, single tenant signal ingestion via Bloo Vantage, configured in two steps from the Crucible UI. Credentials are sealed in Vault and never exposed after initial save.

Connection Configuration & Field MappingV1.0
  • Base URL — Bloo Vantage instance endpoint for this tenant.
  • Cluster / Tenant — Tenant cluster identifier, inherited from Vantage onboarding.
  • Stream — Set to SIGNALS for security alert ingestion. Poll Interval default: 30 seconds.
  • Credentials — Bearer Token sealed in Vault; never displayed after initial save. Rotate Token available from the UI.
  • Field Mapping — $DetectionName → detection_name, $SuspectHost → suspect_host, $DetectionTactic → detection_tactic, + 7 more. Mappings saved per source; take effect on the next poll cycle.

Human Escalation

When a signal requires human judgment, a structured escalation email is sent to a configured analyst distribution list. Analysts act directly from their inbox. All decisions are attributed in the audit trail.

Escalation Email ContentsV1.0
  • Signal summary and detection context
  • Triage verdict and confidence score
  • Top evidence highlights from the investigation
  • One-click deep link to the full Signal Execution View in Crucible

Playbooks

Every incoming signal is automatically matched to an investigation playbook — no manual selection required.

Step Types — Query, Enrich, Triage & RespondV1.0
  • query — Fetch evidence from a connected integration: logs, alerts, events, indicators.
  • enrich — Augment IOCs with threat intelligence: reputation, geolocation, sandbox verdicts.
  • triage — Run specialist Investigation Agents over accumulated evidence; aggregate into a confidence-weighted threat score.
  • respond — Execute a response action against an external system via a configured integration.
Lifecycle — Unpublished/Published & Draft/RetiredV1.0
  • Unpublished — Newly created; not yet active against live signals.
  • Published — The active version currently running against incoming signals.
  • Draft — A pending edit on a live playbook; does not go active until explicitly published.
  • Retired — Soft-deleted; no longer matched to signals. Full version history and rollback retained.

Multi-LLM Provider Support

Crucible is not locked to any single AI provider. Different pipeline stages can use different providers independently — switching requires only a configuration update with no pipeline code changes.

Anthropic · AWS Bedrock · Google Gemini · OpenAI · OllamaV1.0
  • Anthropic Claude — Complex multi-step investigation reasoning; recommended for deep analysis and triage stages.
  • AWS Bedrock — AWS-resident deployments; satisfies existing AWS data residency commitments.
  • Google Gemini — Google Cloud-aligned enterprise deployments.
  • OpenAI — Broad enterprise familiarity; wide model selection across capability tiers.
  • Ollama — Self-hosted, air-gapped, or data-sovereignty-constrained environments; no external API calls.

Authentication & Session Management

Every platform interaction is identity-bound, tenant-scoped, and fully attributable. All pipeline operations require an authenticated session; unauthenticated requests are rejected at the gateway before any investigation or LLM call is initiated.

OAuth 2.0 with PKCE & Email / PasswordV1.0
  • OAuth 2.0 with PKCE — Federated sign-in via Google. Users authenticate through their existing Google account with no password stored on the platform. PKCE is enforced throughout.
  • Email & Password — Available to users provisioned through Bloo Vantage tenant onboarding. Credentials managed within the Bloo Vantage identity layer.
