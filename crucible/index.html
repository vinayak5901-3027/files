Triage Agent

The Triage Agent is the single owner of the full post-playbook investigation pipeline. It orchestrates all specialist Investigation Agents, aggregates their findings into a confidence-weighted threat score, and produces the final verdict and routing action that drives every downstream decision in Crucible. Every run is deterministic, observable, and produces a complete audit record.

Triage Verdict

Every signal is assigned one of three verdicts paired with a confidence score (0–100%):

ActiveA genuine, live threat requiring investigation or response
BenignReal activity but not a threat — expected behaviour that triggered the detection
FalseDetection misfired; no corresponding real-world event found

Confidence-driven routing:

  • Active / high confidence — escalate immediately
  • Benign or False / high confidence — close automatically
  • Any / low confidence — route to human review regardless of verdict

Triage is a continuous re-assessment loop — verdict and confidence update each time a specialist agent returns new evidence.

Structured Output

Every Triage Agent run produces a structured result with four fields:

verdictActive / Benign / False — the primary threat classification
confidence0–100% — how consistently evidence supports the verdict
actionrespond / dismiss / escalate — what the pipeline does next
reasoningHuman-readable explanation of how the verdict was reached

Signal Source

Single source, single tenant signal ingestion via Bloo Vantage, configured in two steps from the Crucible UI. Credentials are sealed in Vault and never exposed after initial save.

Connection Configuration

Each source connection is configured with the following properties:

Base URLBloo Vantage instance endpoint for this tenant
Cluster / TenantTenant cluster identifier — inherited from Vantage onboarding
StreamData stream to poll — set to SIGNALS for security alert ingestion
Poll IntervalHow often Crucible polls for new signals (default: 30 seconds)
CredentialsBearer Token sealed in Vault — never displayed after initial save. Rotate Token available from the UI.

Field Mapping

Raw Vantage source fields are mapped to Crucible's 10 canonical signal fields via a point-and-click UI with live sample values:

$DetectionName→ detection_name
$SuspectHost→ suspect_host
$DetectionTactic→ detection_tactic
+ 7 more fieldsMappings saved per source; take effect on the next poll cycle

Human Escalation

When a signal requires human judgment, a structured escalation email is sent to a configured analyst distribution list or on-call address. Analysts can act directly from their inbox without navigating the platform.

Escalation Email — Contents

  • Signal summary and detection context
  • Triage verdict and confidence score
  • Top evidence highlights from the investigation
  • One-click deep link to the full Signal Execution View in Crucible

Escalation decisions — approved, rejected, or re-investigated — are captured and attributed in the audit trail.

Playbooks

Every incoming signal is automatically matched to an investigation playbook — no manual selection required.

Playbook Step Types

Five step types define what each playbook step does:

queryFetch evidence from a connected integration — logs, alerts, events, indicators
enrichAugment IOCs with threat intelligence — reputation, geolocation, sandbox verdicts
triageRun specialist Investigation Agents over accumulated evidence; aggregate into a confidence-weighted threat score
respondExecute a response action against an external system via a configured integration

Playbook Lifecycle

Playbooks follow a controlled four-state lifecycle:

UnpublishedNewly created — not yet active against live signals
PublishedThe active version currently running against incoming signals
DraftA pending edit on a live playbook — does not go active until explicitly published
RetiredSoft-deleted; no longer matched to signals. Full version history and rollback retained.

Multi-LLM Provider Support

Crucible is not locked to any single AI provider. Different pipeline stages can use different providers and models independently — switching requires only a configuration update with no pipeline code changes.

Supported Providers

Anthropic ClaudeComplex multi-step investigation reasoning; recommended for deep analysis and triage stages
AWS BedrockAWS-resident deployments; satisfies existing AWS data residency commitments
Google GeminiGoogle Cloud-aligned enterprise deployments
OpenAIBroad enterprise familiarity; wide model selection across capability tiers
OllamaSelf-hosted, air-gapped, or data-sovereignty-constrained environments; no external API calls

Authentication & Session Management

Every platform interaction is identity-bound, tenant-scoped, and fully attributable. All pipeline operations require an authenticated session; unauthenticated requests are rejected at the gateway before any investigation or LLM call is initiated.

OAuth 2.0 with PKCE

Federated sign-in via Google as the identity provider. Users authenticate through their existing Google account with no password stored on the platform. PKCE is enforced throughout to protect the authorisation flow against interception.

Email & Password

Available to users provisioned through Bloo Vantage tenant onboarding. Credentials are managed within the Bloo Vantage identity layer; Crucible inherits the authenticated session on successful sign-in.

Deployment & Security

Enterprise-grade security,
on your infrastructure.

Both Bloo products are self-hosted on Kubernetes on AWS or Azure. Customer data — including investigation records and AI call content — never leaves your environment.

Self-Hosted KubernetesV1.0

Both products deploy on Kubernetes on AWS or Azure. Customer data — including investigation records, query payloads, and AI call content — does not leave the customer's environment.

Air-Gapped DeploymentsV1.0

Air-gapped deployments are supported via Ollama for LLM execution. No external AI API calls required — the entire triage pipeline runs within the customer perimeter.

Secrets & AuditV1.0

All integration and LLM credentials are sealed in enterprise-grade secrets vaults. Session tokens use short TTLs with silent refresh. Every action and AI reasoning step is written to an append-only audit trail that cannot be modified or deleted.
