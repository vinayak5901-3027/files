Pipeline Factory — 5-Stage Query Intelligence Pipeline

The core execution engine of SynthAI. Every natural-language question flows through five deterministic, observable stages. Live progress events are emitted over WebSocket at each stage transition.

Stage 1–5: Decomposer · Query Generator · Executor · Analyser · SynthesizerV1.0
  • Stage 1 — Decomposer: Breaks the question into independent sub-questions, each mapped to a specific DataFabric stream with a defined time range and candidate fields.
  • Stage 2 — Query Generator: Converts each sub-question into validated SQL. Guardrails block DDL statements, unbounded time scans, and cross-scope joins.
  • Stage 3 — Executor: Runs all validated sub-queries in parallel via the DataFabric API, each scoped to the tenant's Scope.
  • Stage 4 — Analyser: Analyses each result set independently — extracting key findings and flagging notable values. Per-stream severity ratings are assigned.
  • Stage 5 — Synthesizer: Produces a structured response streamed to the UI — direct answer, evidence cards, confidence rating, severity assessment, suggested next steps, and exact SQL for each sub-query.

Recommendation Control

Users can toggle the Recommendations section of any pipeline response on or off per session.

Enabled & DisabledV1.0
  • Enabled — Synthesizer includes a recommendations block; follow-up investigation tasks may be spawned from the response.
  • Disabled — Recommendations block omitted entirely; no follow-up tasks are spawned. Toggle state is preserved per session.

Token & Usage Display

Every LLM call across all five pipeline stages is logged in real time. Live usage events are pushed to the browser as each stage completes — costs update while the pipeline runs, not as a post-hoc summary.

Real-Time Cost TrackingV1.0
  • Provider, model, input tokens, output tokens, estimated cost, and latency — all logged per stage.
  • Live usage events pushed to the browser as each stage completes.
  • Costs update while the pipeline runs — not as a post-hoc summary.

Authentication & Session Management

Every query is identity-bound and tenant-scoped. Sessions use short-lived access tokens refreshed transparently — unauthenticated requests are rejected at the gateway before any data or LLM call is made.

OAuth 2.0 & Email / PasswordV1.0
  • OAuth 2.0 — Federated sign-in via Google. Users authenticate through their existing Google account with no password stored on the platform.
  • Email & Password — Available to users provisioned through Bloo DataFabric tenant onboarding. Credentials managed within the Bloo DataFabric identity layer.

LLM Licence Management

Self-service LLM connection management — organisations register, validate, and manage their own provider credentials without routing any key through the application database.

Connection Validation & Credential StorageV1.0
  • Connection Validation — Every saved connection is immediately test-fired. Only marked verified after a successful live test. Failed keys surfaced at save time with a clear error.
  • Credential Storage — API keys are sealed in an enterprise-grade secrets vault at the moment of entry. Keys are never saved in the application database and cannot be read back. Only non-sensitive metadata persisted. Keys replaceable via Rotate Key action.

Multi-LLM Provider Support

SynthAI supports four LLM provider families. Different scopes within the same organisation can use different providers simultaneously. Switching providers requires no pipeline code changes.

Anthropic · AWS Bedrock · OpenAI · OllamaV1.0
  • Anthropic — Long-context reasoning; recommended for complex multi-stream investigation questions.
  • OpenAI — Broad enterprise familiarity; wide model choice across capability and cost tiers.
  • AWS Bedrock — AWS-resident deployments; satisfies existing data residency and sovereignty commitments.
  • Ollama — Fully self-hosted; air-gapped or data-sovereignty-constrained environments; no external API calls.
